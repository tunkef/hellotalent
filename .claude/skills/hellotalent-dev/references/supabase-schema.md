## Core Tables

### candidates
- id (GENERATED BY DEFAULT — allows upserts)
- user_id (FK → auth.users)
- full_name, email, telefon
- is_approved, is_active
- avatar_url, cv_url, cv_filename, cv_uploaded_at
- hide_from_current_employer (boolean, default false)
- updated_at (auto-trigger)
- RLS: candidates scoped to own data

### Child Tables (candidate profile)
- candidate_experiences
- candidate_education
- candidate_certificates
- candidate_languages
- candidate_target_roles
- candidate_work_preferences
- candidate_brand_interests
- candidate_location_preferences (+ districts)

### hr_profiles
- id (FK → auth.users — cannot write without valid auth UUID)
- RLS: self-access (SELECT/INSERT/UPDATE own row)
- INSERT policy enforces auth.uid() = id

### Company Ecosystem (Phase A)
- companies (61 rows live)
- brands (96 rows live)
- candidate_company_follows
- RLS enabled on all three

## Storage
- **Bucket:** cvs
- **Avatar path:** avatars/{user_id}.{ext}
- **CV path:** cv/{user_id}/cv.{ext}
- Single active CV per user, single active avatar per user

## Key RPC
- `save_candidate_profile` — single RPC for candidate profile persistence

## Critical Rules
- `candidates.id` is GENERATED BY DEFAULT (not ALWAYS) — upserts work
- Upserts need UNIQUE INDEX on target column before onConflict
- Service role key bypasses RLS (admin only)
- hr_profiles FK constraint means no direct DB write without valid auth UUID
- Email source of truth: Supabase Auth (not candidates table)
